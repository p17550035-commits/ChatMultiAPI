# Heartbeat / Pulse Monitoring Subsystem

## 1. Purpose
Provide continuous, periodic communication between every module and its router, enabling:
- Real-time health monitoring
- Automatic failure detection
- Automatic redundancy switching
- Automatic recovery attempts
- Automatic notification triggers

This subsystem acts as the circulatory system of the closed-loop architecture.

---

## 2. Heartbeat Flow (Per Module)

Each module sends a heartbeat packet to its router at fixed intervals.

Heartbeat packet includes:
- module_id
- timestamp
- status (OK, DEGRADED, OFFLINE)
- last_error
- uptime
- load metrics (optional)
- correlation_id

Routers forward heartbeat packets to:
- Layer wiring hub
- Master wiring hub (primary)
- Redundant wiring hub (secondary)

---

## 3. Failure Detection Logic

If a module misses X heartbeats:
- Router marks module as DEGRADED
- Hub marks module as OFFLINE
- Hub triggers:
  - Notification subsystem
  - Calendar subsystem (optional scheduling)
  - Redundancy subsystem (breaker/track switch)

If heartbeat resumes:
- Router marks module as RECOVERED
- Hub marks module as ONLINE
- Notification subsystem sends recovery alert

---

## 4. Redundancy Wiring Integration

Heartbeat is the trigger for redundancy switching.

Flow:
1. Module heartbeat stops
2. Router reports failure to hub
3. Hub signals master wiring hub
4. Master hub flips breaker → activates secondary orchestration line
5. Secondary line attempts reconnection
6. Secondary line begins pinging module
7. If module responds:
   - Hub marks module as ALIVE
   - Notification subsystem sends recovery alert
8. If module does not respond:
   - Hub marks module as DEAD
   - Notification subsystem sends critical alert
   - Calendar schedules recovery window

---

## 5. Notification Triggers

Heartbeat failure triggers:
- Text message
- Email
- Push notification
- Emergency alert (for critical modules)
- Optional automated call

Heartbeat recovery triggers:
- “Module restored” text/email/push/call
- Status report including:
  - downtime duration
  - recovery path (primary or secondary line)
  - module health metrics

---

## 6. Router Responsibilities

Routers must:
- Receive heartbeat packets
- Validate packet structure
- Timestamp arrival
- Detect missed intervals
- Forward status to hubs
- Provide structured logs for toaster LLM debugging

Routers NEVER talk directly to other modules.

---

## 7. Hub Responsibilities

Layer hubs:
- Track heartbeat status for their module
- Report status to master hub
- Trigger redundancy switching
- Trigger notifications
- Trigger calendar scheduling

Master hub:
- Maintains global heartbeat map
- Controls breaker/track switching
- Coordinates recovery attempts

Redundant hub:
- Takes over routing during primary failure
- Performs reconnection pings
- Reports recovery or death

---

## 8. Closed-Loop Integration

Heartbeat ties the entire closed-loop system together:

- Foundation Forge sends heartbeat → Orchestration Cortex monitors
- Functional layers send heartbeat → Hubs track status
- Notification layer reacts to heartbeat changes
- Calendar layer schedules recovery windows
- Redundancy wiring switches based on heartbeat loss
- Orchestration Cortex adjusts routing based on heartbeat map

This makes the system:
- self-aware
- self-correcting
- self-healing
- self-notifying
- self-routing

---

## 9. Summary

The heartbeat subsystem:
- Detects failures
- Triggers redundancy switching
- Attempts reconnection
- Confirms recovery
- Notifies you
- Schedules fixes
- Keeps the closed-loop architecture alive

It is now a core part of the Frankenstein Meta-OS blueprint.
